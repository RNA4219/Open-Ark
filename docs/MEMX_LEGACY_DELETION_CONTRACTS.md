# 旧記憶実装 削除候補 契約表

## 1. 位置づけ

本書は、Open-Ark における旧記憶実装の削減判断に先立ち、
「何が残っていれば互換とみなすか」を定義する補助文書である。

主目的は次の 3 つとする。

1. 削除候補ごとの最低保証を明確にする
2. 旧実装と memx 代替経路の同値判定観点を固定する
3. 削除前に必要なテスト観点を整理する

本書は削除許可書ではない。
削除可否の最終判断は、各契約に対応するテストが成立した後に行う。

---

## 2. 対象

削除候補として扱う対象は次の 5 系統とする。

1. `entity_memory_manager`
2. `episodic_memory_manager`
3. `rag_manager`
4. `dreaming_manager_sync` / `dreaming_manager` 周辺
5. `motivation_manager_sync` / `motivation_manager` 周辺

次は現時点で削除対象に含めない。

- `memory_manager`
- `entity_tools`
- `memory_tools`
- `local_adapter`

---

## 3. 契約表

### 3.1 `entity_memory_manager`

**目的**

エンティティ知識を保存、更新、参照、検索できること。

**最低保証**

- エンティティ名をキーに知識を保存できる
- 既存エントリを追記または上書きできる
- 単一エントリを取得できる
- 一覧取得できる
- キーワード検索で関連エントリを返せる
- room をまたいで混線しない

**代替経路**

- `memx_ingest(store="knowledge")`
- `memx_search`
- `memx_show`

**同値判定**

- 保存後に再取得できる
- 検索語に対して対象エンティティが見つかる
- 別 room からは見えない

**削除可条件**

- 上記を memx 経路で満たす
- `entity_tools` が旧実装なしでも成立する
- `disabled` 状態で主要導線が維持される

---

### 3.2 `episodic_memory_manager`

**目的**

出来事や進捗を時系列のエピソードとして保持し、後で文脈参照できること。

**最低保証**

- エピソードを保存できる
- 最新日時または時系列順の参照ができる
- 過去エピソードを文脈として取得できる
- room ごとに独立している

**代替経路**

- `memx_ingest(store="journal")`
- `memx_search`
- `memx_recall(mode="recent" または "related")`

**同値判定**

- 保存した出来事が後で取得できる
- 時系列文脈に反映される
- 他 room の出来事が混ざらない

**削除可条件**

- journal 系の保存と参照が memx 側で再現できる
- 旧エピソード参照が read-only 不要と判断できる
- `disabled` 状態で周辺導線が壊れない

---

### 3.3 `rag_manager`

**目的**

質問や状況に対して関連記憶、関連文脈を検索できること。

**最低保証**

- クエリ入力に対して関連文脈を返せる
- 結果が空でも異常終了しない
- 現在ログや知識ベース由来の文脈取得ができる
- room ごとに検索対象が分離される

**代替経路**

- `memx_recall`
- `memx_search`

**同値判定**

- 同じクエリで関連記憶が返る
- 検索失敗時に graceful に空結果または代替結果になる
- 知識検索の用途を memx 側で満たせる

**削除可条件**

- `memory_tools` / `knowledge_tools` が旧 RAG 依存なしで通る
- `disabled` 状態で主要検索導線が維持される
- import 欠落シミュレーションでも起動が落ちない

---

### 3.4 `dreaming_manager_sync` / `dreaming_manager` 周辺

**目的**

夢、洞察、発見を保存し、後続文脈で再利用できること。

**最低保証**

- 洞察を保存できる
- 最近の洞察を再取得できる
- 夢、内省由来の記録が journal 系文脈として扱える
- 本体機能が同期失敗で落ちない

**代替経路**

- `memx_ingest(store="journal")`
- `memx_recall`

**同値判定**

- 保存した洞察が後で取得できる
- journal 系検索で見つかる
- 同期失敗時も dreaming 本体が継続する

**削除可条件**

- 同期責務だけを切り離しても dreaming 本体は生きる
- memx journal で保存、再取得が成立する
- `disabled` 状態で夢生成導線が壊れない

---

### 3.5 `motivation_manager_sync` / `motivation_manager` 周辺

**目的**

未解決の問いや内的状態を保持し、必要時に参照、更新できること。

**最低保証**

- 未解決問いを追加できる
- 内部状態を取得できる
- 問いを解決済みにできる
- interaction 更新で異常終了しない
- room ごとに独立している

**代替経路**

- `memx_ingest(store="short")`
- `memx_recall`
- `memx_resolve`

**同値判定**

- 追加した問いが後で見つかる
- 解決後に状態が変わる
- 他 room に漏れない

**削除可条件**

- 問い管理の保存先が memx short へ移っても成立する
- motivation 本体が同期責務なしで動く
- `disabled` 状態で接触判定や内部状態取得が壊れない

---

## 4. 共通テスト観点

全候補について、少なくとも次の観点を確認しなければならない。

- 保存
- 再取得
- 検索
- 更新または解決
- room 分離
- fallback
- `read_only`
- `disabled`
- import 欠落シミュレーション
- Phase 2 / Phase 3 検収への影響なし

---

## 5. 推奨テスト層

削除判断前に、次の 3 層を用意することを推奨する。

### 5.1 契約テスト

各削除候補について、「何を返せば互換とみなすか」を固定する。

### 5.2 旧実装 vs 代替経路の同値テスト

同じ入力に対して、旧実装と memx 代替経路が意味的に同等の結果を返すことを確認する。

### 5.3 削除シミュレーションテスト

対象 import を意図的に失敗させても、代替導線または明示エラーで安全に検知できることを確認する。

---

## 6. 削除判断の前提

削除してよいと判断できるのは、少なくとも次を満たした対象に限る。

1. 契約テストが存在する
2. 代替経路の同値テストが通る
3. `disabled` 状態で主要導線が通る
4. import 欠落シミュレーションでも致命障害にならない
5. Phase 2 / Phase 3 の検収結果を悪化させない

上記のいずれかを満たさない場合、その対象は削除してはならない。
